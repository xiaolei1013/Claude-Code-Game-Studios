## Overview

*Lantern Guild* is an idle/incremental game where the player runs a fantasy hero guild, assigns class-based formations to dungeons, and returns to accumulated loot that funds stronger recruits. Session-based: 2-5 min × 2-4 times per day. Idle games are **content-light but system-heavy** — this index enumerates 30 MVP+polish systems plus 2 V1.0 stubs.

The design emphasis is on the **Offline Progression Engine** and **Class-vs-Enemy Matchup Resolver** — these two systems carry the emotional weight of the concept (recruit-to-overpower loop + strategic formation choice) and are where tuning failure would invalidate the core loop. Both are flagged as high-risk and sit in the first half of the design order.

All 30 systems serve the four pillars:
1. **Respect the Player's Time** — Offline Progression, Return-to-App, no-FOMO economy design
2. **Every Class Feels Distinct** — Class DB, Matchup Resolver, Formation Assignment, Roster UI
3. **Matchup Is a Decision, Not a Reflex** — Matchup Resolver, Formation Assignment, Matchup Assignment Screen
4. **HD-2D Pixel Pride** — HD-2D Rendering Pipeline, VFX, UI Framework (parchment theme)

## Dependency Map

### Foundation Layer (no dependencies)

1. **Game Time & Tick System** — provides authoritative `now_ms()` and session-boundary timestamps; everything idle-related depends on this
2. **Data Loading System** — single loader pattern for `.tres` game data (classes, enemies, biomes, dungeons)
3. **Scene/Screen Manager** — SceneTree helper for screen transitions
4. **UI Framework / Theme** — parchment theme, tap-target enforcement, touch/mouse parity layer (art bible binds it)

### Core Layer (depends on foundation)

1. **Save/Load System** — depends on: Data Loading. Includes anti-tamper; everything mutable depends on this.
2. **Economy System** — depends on: Game Time, Save/Load
3. **Hero Class Database** — depends on: Data Loading
4. **Enemy Database** — depends on: Data Loading
5. **Biome & Dungeon Database** — depends on: Data Loading, Enemy DB
6. **Audio System** — depends on: Scene Manager

### Feature Layer (depends on core)

1. **Hero Roster** — depends on: Class DB, Save/Load
2. **Class-vs-Enemy Matchup Resolver** — depends on: Class DB, Enemy DB
3. **Combat Resolution System** — depends on: Roster, Matchup Resolver
4. **Offline Progression Engine** ⚠️ — depends on: Game Time, Combat Resolution, Economy
5. **Dungeon Run Orchestrator** — depends on: Combat Resolution, Dungeon DB, Offline Engine
6. **Recruitment System** — depends on: Class DB, Roster, Economy
7. **Hero Leveling System** — depends on: Roster, Economy
8. **Floor/Biome Unlock System** — depends on: Dungeon DB, Save/Load
9. **Formation Assignment System** — depends on: Roster, Dungeon DB

### Presentation Layer (UI screens + render pipeline)

3. **Recruit Screen** — depends on: Recruitment, UI Framework
4. **Roster / Hero Detail Screen** — depends on: Roster, UI Framework

## Circular Dependencies

**None found.** The one near-cycle worth noting:

- **Offline Engine ↔ Dungeon Run Orchestrator**: These conceptually wrap each other (Offline Engine calls Run Orchestrator for each simulated run; Run Orchestrator returns results to Offline Engine). **Resolution**: Run Orchestrator exposes a pure-function `simulate_run(formation, dungeon, duration) -> RunResult` contract. Offline Engine is a caller, not a peer. No circularity at the code level.

## High-Risk Systems

| System | Risk Type | Risk Description | Mitigation |
|--------|-----------|-----------------|------------|
| **Offline Progression Engine** (#12) | Tech + Design + Scope | Math tuning is subjective; "feels fair" is uncalibrated until real users play; bugs here permanently break player trust | Prototype first after foundations (design order #11); use `/prototype` before first real story; budget 2-3 tuning passes |
| **Class-vs-Enemy Matchup Resolver** (#10) | Design | Too subtle → invisible to players; too strong → rock-paper-scissors degenerates strategy. Must feel like a decision, not a lookup | Prototype matchup multipliers with 3 classes × 8 enemies; playtest at minimum viable scale before committing formulas |
| **Save/Load with anti-tamper** (#3) | Security | Idle games are the #1 target for save-editing and time manipulation; compromise here = player cheats self out of the progression loop | Consult security-engineer during design; design checksums + timestamp validation from day one; do not ship cloud save until V1.0 |
| **Return-to-App Screen** (#20) | UX | First 30s of every session; cozy-game UI most commonly fails here. Art bible flagged legibility-over-atmosphere as binding | Prototype with a first-session usability test; hold ux-designer + art-director joint review before committing to final layout |
| **HD-2D Rendering Pipeline** (#26) | Tech | Godot 4.6 `Compositor`/`CompositorEffect` is post-LLM-training-cutoff; Shader Baker + glow-blending behavior changed in 4.6; SMAA is 4.5+ | Verify APIs against `docs/engine-reference/godot/breaking-changes.md` before story work; ship MVP without shader pass if needed (art bible marks this as Vertical Slice priority deliberately) |
