## Overview

The Economy autoload owns gold balance, the per-floor first-clear monotonic credit
ledger, and the cost curves that gate recruit / level-up / formation decisions.
It subscribes to `tick_fired` at 20 Hz for foreground drip and exposes a
closed-form `compute_offline_batch(tick_budget)` for the Offline Progression
Engine. All 26 tuning knobs live in `assets/data/config/economy_config.tres`
(EconomyConfig extends GameData per ADR-0011 pattern); zero hardcoded balance
values in code.

Economy is the highest-stakes balance system in the project — per
`.claude/docs/technical-preferences.md`, offline-progression math is the
project's #1 risk area, and a bug here breaks player trust in a single visible
session.

## Governing ADRs

| ADR | Decision Summary | Engine Risk |
|-----|------------------|-------------|
| ADR-0013 — Economy State + Cost Curves + Offline Batch Contract | 7-method public API; persisted state shape; recruit/level cost curves; closed-form drip; 4 forbidden patterns (hardcoded-balance, losing-run-read, offline-replay-emit, negative-spend) | LOW |
| ADR-0014 — Offline Replay Batch Chunking + RunSnapshot Schema | Adaptive 12 ms-budget chunking; main-thread `await process_frame` yield; signal suppression during replay + aggregate emit after | MEDIUM |
| ADR-0011 — Resource Schemas Core Databases | EconomyConfig.tres schema + DataRegistry-loaded tuning knobs | LOW |
| ADR-0005 — Time System Dual-Clock Contract | Economy subscribes `tick_fired` only; never reads `_process(delta)` | LOW |
| ADR-0003 — Autoload Rank Table | Economy at rank 3, after TickSystem (0) / DataRegistry (1) / SaveLoadSystem (2) | LOW |
| ADR-0002 — LOSING First-Clear Reclaimable on Win | `floor_clear_bonus_credited: Dictionary[int,int]` monotonic ledger field | LOW |
| ADR-0004 — Save Envelope + HMAC Scheme | Economy participates in save consumer contract via `get_save_data` / `load_save_data` | MEDIUM |

**Highest-risk ADR**: ADR-0014 (offline batch chunking). MEDIUM risk —
adaptive time-budgeted chunking with `await get_tree().process_frame` yield is
well-supported in 4.6 but fiddly to validate at the 16 ms min-spec ceiling.

## Cross-Cutting Inputs (received from)

- `TickSystem` (Foundation) — `tick_fired(n)` 20 Hz signal
- `DataRegistry` (Foundation) — resolves `EconomyConfig`, hero class tier lookup
- `SaveLoadSystem` (Foundation) — invokes `get_save_data` / `load_save_data` per ADR-0004 consumer contract

## Cross-Cutting Outputs (consumed by)

- Recruitment (Feature) — calls `recruit_cost(class_id, copies_owned)` + `try_spend(amount, reason)`
- HeroLeveling (Feature) — calls `level_cost(tier, level)` + `try_spend`
- DungeonRunOrchestrator (Feature) — calls `try_award_floor_clear(floor_index, bonus)`
- OfflineProgressionEngine (Feature) — calls `compute_offline_batch(tick_budget)`
- All Presentation screens — subscribe `gold_changed(new_balance, delta, reason)`
