## Codebase Patterns

- Test runner invocation (macOS, Godot mono build):
    /Applications/Godot_mono.app/Contents/MacOS/Godot --headless \
      -s addons/gdUnit4/bin/GdUnitCmdTool.gd --ignoreHeadlessMode \
      -a tests/unit/<dir-or-file>
  The wrapper `tests/gdunit4_runner.gd` exists but the direct CmdTool form is
  what CI uses. `--ignoreHeadlessMode` is REQUIRED — without it the runner
  exits 103 (headless-mode unsupported guard).
- Audit predicate (per EPIC technical considerations):
    grep -nE "^func [a-z]" src/core/<system>/<file>.gd   # public funcs
    grep -rn "<func_name>" tests/                         # references

**Learnings for future iterations:**
- For accessor-style functions (`set_*` DI seams) the canonical happy+edge pair
  is "replaces_internal_field" (happy) + "survives_ready_call" (edge —
  injection-before-_ready survives the lazy-default branch). When auditing
  other autoloads' DI seams (TickSystem, SaveLoadSystem, SceneManager), look
  for this paired pattern in autoload_skeleton_*.gd files and accept it as
  full coverage.
- For functions with implementation branches gated by data-registry state
  (e.g., `resolve_floor_by_snapshot_id` only returns non-null if "floors" were
  a registered category — it isn't), BOTH observable-behavior branches (empty
  + unknown) ARE the full coverage surface. A "happy path" doesn't have to
  return a non-null/non-error value; it just has to assert the documented
  return shape. This is consistent with FR-2 ("documented return shape /
  behavior").
- Heavy-coverage files like dungeon_run_orchestrator (42 test files) only
  need an audit pass — searching `grep -rn "<func_name>" tests/` and
  spot-checking the assertion shape suffices. Don't be tempted to "improve"
  test names or refactor existing tests; that's out of scope per FR-8 spirit
  ("non-regression in baseline" implies "don't touch passing tests").

  accessor invariant (no state mutation). Mirrors the
  `recruit_cost_null_config_returns_minus_one` add_child + null-override
  pattern from economy_recruit_cost_test.gd.
- Audit predicate refinement: when a public func has 0 direct test
  references (`grep -rn "<func_name>" tests/` returns nothing matching the

  value is consumed elsewhere. The existing economy/ suite extensively

  never `economy.get_config()` itself — pure accessor public surface still
  needs at least one direct call site under assertion.

  "Production callers MUST NOT set this"); coverage via every
  compute_offline_batch happy test counts because the seam IS the input
  channel for those tests. Don't add a separate dedicated test for DI seams
  whose only legitimate caller is the test harness itself.

**Learnings for future iterations:**
- The graceful-degrade pattern in enemy_database_autoload_test.gd
  (push_warning + return when DataRegistry is not READY) is correct for
  integration tests but produces tests that *trivially pass* without
  assertions in environments where the DataRegistry has 0 enemy .tres

  separate happy-path coverage.

  resolve cast; `get_all_ids` = filter-and-sort over get_all_by_type) are
  a recurring pattern across rank-5 autoloads (EnemyDatabase,
  HeroClassDatabase, BiomeDungeonDatabase). Their coverage bar is met by
  asserting the *cast/filter behavior* (returns null/empty on miss; returns
  typed result + sort order on hit), not by re-asserting DataRegistry's
  own contracts.

**Learnings for future iterations:**
- The `OS.is_debug_build()` guard branch on debug helpers is NOT directly
  testable in gdunit4 (the test runner IS a debug build by definition; no
  hook lets us simulate `OS.is_debug_build() == false` without source-level
  refactor). Document the production-blocked branch as covered-by-construction
  and focus tests on the in-debug-build mutating path.
- When auditing autoloads with debug/test API surface (`debug_*` methods),
  expect them to be consumed as integration-test setup but NOT to have direct
  unit-test coverage. Future US-NNN audits for autoloads with debug helpers
  (HeroRoster.debug_*, FormationAssignment.debug_*, Recruitment.debug_*) will
  likely surface the same gap shape — reuse the helper-test file pattern from
  debug_helpers_test.gd.
- Mutator helpers (debug_*) need **paired happy + bypass-edge** tests:
  happy = "writes value, accessor reflects" (the contract);
  edge1 = "bypasses an invariant the production path enforces" (e.g., monotone
    R4 → debug can decrease; over-range clamp → debug accepts 999 unclamped);
  edge2 = "skips a validation the production path requires" (e.g., biome
    availability check → debug accepts unknown biome).
  This shape captures WHY the helper is dangerous in production code-paths.
